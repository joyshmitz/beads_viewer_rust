# `bvr` robot-contract consumer examples

This directory shows the architectural thesis that issue #12 landed on: `bvr --robot-*` primitives are **integration contracts**, and human-facing lenses and downstream integrations live downstream of them. There is no HTML renderer inside `bvr` here — every example reads the stable JSON envelopes and decides for itself how to render for its reader.

## Layout

```
examples/contract-consumers/
├── triad.sh                    # orchestrator: runs the three primitives, caches JSON
├── economics.sample.json       # overlay template for --robot-economics (copy to .bv/economics.json)
├── lenses/
│   ├── engineer/brief.sh       # terminal brief for a developer
│   ├── owner/brief.sh          # delivery posture for a delivery lead
│   ├── investor/brief.sh       # finance-style view with provenance
│   └── erp/adapter.jq          # normalizer into a finance-shaped schema
├── portfolio/
│   └── rollup.sh               # aggregator across N projects
└── README.md                   # this file

.bv/                            # local-only (gitignored)
├── economics.json              # your overlay copy (required for --robot-economics)
└── runs/                       # cached --robot-* outputs (regenerated by triad.sh)
    ├── overview.json
    ├── delivery.json
    └── economics.json
```

## Quick Start

All commands run from the repo root. `bvr` discovers `.beads/` under the current repo by default; pass `--beads-file <path>` to override (see main `README.md` → Input Model).

```bash
# 1. Build bvr
cargo build --release --bin bvr

# 2. Prepare the local economics overlay (one-time)
mkdir -p .bv && cp examples/contract-consumers/economics.sample.json .bv/economics.json
# (edit .bv/economics.json with your own rates)

# 3. Generate the triad (writes into .bv/runs/)
examples/contract-consumers/triad.sh

# 4. Run lenses over the cached JSON
examples/contract-consumers/lenses/engineer/brief.sh     # for yourself
examples/contract-consumers/lenses/owner/brief.sh        # for a delivery lead
examples/contract-consumers/lenses/investor/brief.sh     # for finance
jq -f examples/contract-consumers/lenses/erp/adapter.jq --arg project bvr .bv/runs/economics.json
```

Portfolio rollup across several repos:

```bash
examples/contract-consumers/portfolio/rollup.sh /path/to/proj-a /path/to/proj-b /path/to/proj-c
```

The envelopes are JSON by default; the same primitives also emit TOON via `--format toon` (see main `README.md` → Output formats) if a downstream consumer prefers that transport.

## What each lens does with the same facts

All four consumers work off the same cached envelopes (one `triad.sh` run, one shared `data_hash`). The difference is in **how** they speak about it.

- **Engineer** — action-oriented: next move, top blockers by `dependents_count`, which guards tripped. Compact, with a ready-to-paste `claim_command`.
- **Owner** — aggregate-oriented: flow distribution, urgency cohorts, milestone pressure. No claim commands; a delivery lead is not claiming tasks from this view, though milestone pressure still names specific issues when due dates are tight.
- **Investor** — numbers- and provenance-oriented: input assumptions, projections, boolean guards relabeled as `TRIPPED` / `ok`, and a provenance footer grouping `data_hash` + `overlay_hash`. Deliberately no "on track" / "at risk" labels — the downstream consumer picks its own thresholds.
- **ERP** — transport-shape-oriented: a normalized JSON structure with snake_case fields, currency at the top level, `guards_tripped` as a flat list, provenance as its own block. A finance system ingests this JSON without needing to know about `beads` or `bvr`.

## Why this validates the issue #12 architecture

If economics or delivery had shipped as an HTML primitive inside `bvr` (as the original `AUDIENCE_EXPORT_THINKING_LOG` sketched), each of these four consumers would have to either parse HTML or maintain a parallel implementation of the same computation. Instead they all live off one payload, and the next reader — a Slack digest, a compliance archive, a CI gate — drops in without touching `bvr`.

The portfolio rollup makes this explicit: N projects × `--robot-economics` becomes one aggregate through plain arithmetic over the payloads. HTML lenses do not compose that way.

## What these examples are **not**

- Not production-ready. Shell + `jq` here is pattern demonstration; a real finance integration usually writes the adapter in Go / Python / TypeScript with schema validation and retry logic.
- Not the canonical lenses. "Investor" here is one convention; a real investor view may need fields (runway, burn multiple, ratio) that the current contract does not expose. That's fine — a new lens consumer adds those computations on its own side rather than forking `bvr`.
- Not ERP integrations. `erp/adapter.jq` returns a sample shape; a real integration maps to concrete SAP / NetSuite / QuickBooks fields, or whatever the integrator accepts.

## Directions to extend

A map of where these examples can grow:

- **Developer** (engineer): add a git-blame-adjusted prompt that auto-tags issues where human attention has been sparse.
- **Delivery lead** (owner): a CSS-only HTML PMO dashboard, rendered from `delivery.json` + `economics.json` via a nightly cron.
- **Investor / finance / board**: a PDF report from the same two files, with `overlay_hash` as the audit anchor.
- **ERP**: a real adapter into the format that SAP S/4HANA / NetSuite / an internal OLAP system will accept.
- **Compliance / audit**: `git checkout <ref>` followed by a re-run of `triad.sh` reconstructs state at any historical date.
- **Multi-agent swarm coordinator**: reads `cost_of_delay[]` as a circuit breaker and reassigns agent workloads by unblock priority.
